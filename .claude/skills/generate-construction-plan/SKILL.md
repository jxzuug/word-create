# 施工组织设计 / 服务方案 Word 文档生成

## 工作流程

### 步骤 0: JSON 编码规范与验证（必须遵守）

【严重】在构建 JSON 数据时，必须严格遵守以下编码规范。违反此规范将导致 JSON 解析失败、后续追加和文档生成全部中断。

#### 规则1：中文文本中绝对禁止使用 ASCII 双引号

中文文本中需要引号时，必须使用以下任意一种方式：
- **推荐**：使用 `“`（左双引号"）和 `”`（右双引号"）
- **备选**：使用方角引号 `「` 和 `」`
- **禁止**：在 JSON 字符串内使用未转义的 ASCII 双引号 `"` (U+0022)

```json
// ❌ 错误：中文"xxx"会破坏JSON结构
{ "text": "遵循"统一管理、分区施工"的原则" }

// ✅ 正确：使用 Unicode 转义
{ "text": "遵循“统一管理、分区施工”的原则" }

// ✅ 正确：使用方角引号
{ "text": "遵循「统一管理、分区施工」的原则" }
```

#### 规则2：每次写入 JSON 后必须立即验证

使用 `Write` 工具写入 JSON 文件后，**必须立即**运行验证命令：

```bash
node -e "JSON.parse(require('fs').readFileSync('temp.json','utf8')); console.log('JSON valid')"
```

如果验证失败，参考下方自动修复脚本。

#### 规则3：JSON 修复脚本模板

当 JSON 解析失败时，使用以下脚本自动识别并修复中文文本中被错误编码的 ASCII 引号：

```javascript
// fix-quotes.js — 自动修复 JSON 中文文本中的 ASCII 引号
const fs = require('fs');
let content = fs.readFileSync('temp.json', 'utf8');
const CJK = /[一-鿿㐀-䶿A-Za-z0-9]/;

let result = '';
for (let i = 0; i < content.length; i++) {
    const ch = content[i];
    const prev = i > 0 ? content[i-1] : '';
    const next = i < content.length - 1 ? content[i+1] : '';
    
    if (ch === '"' && !/[{\[,:}\s\n\r\t]/.test(prev) && !/[}\],:\s\n\r\t]/.test(next)) {
        // 内部引号：前面有中文 → 右引号；否则 → 左引号
        result += CJK.test(prev) || /[、，。；：）\)]/.test(prev) ? '”' : '“';
    } else {
        result += ch;
    }
}
fs.writeFileSync('temp.json', result, 'utf8');
// 验证修复结果
JSON.parse(result);
console.log('Fixed and validated successfully');
```

使用方法：`Write` 写 `fix-quotes.js` → `node fix-quotes.js`

【重要】在生成任何 JSON 数据之前，先理解以上规则。每轮写入 temp.json 后必须验证 JSON 合法性。

### 步骤 1: 接收项目信息

用户会直接提供项目信息。典型格式：

```
项目名称：XXX
项目概况/服务内容：XXX
章节结构：
第一章、XXX
第二章、XXX
...
需要输出：XX页 或 约XXX字
```

**无需逐项询问** — 用户给什么就用什么。如果用户信息不完整，仅针对缺失的关键信息进行追问（如没有项目名称或章节结构时）。

#### ⚠️ 内容覆盖完整性优先于页数目标（核心原则）

当用户同时提供了**工程量清单（含具体施工项）**和**页数目标**时，可能产生冲突：
- 工程量清单有 N 项具体设备/工序需要编写施工方案
- 页数目标算出的子章节数不够覆盖所有清单项
- **错误做法**：压缩或跳过部分清单项的施工描述来凑页数
- **正确做法**：先确保每项清单内容都有施工方法，页数超出预期则告知用户

**分析方法：**

1. 提取工程量清单中的施工项数（统计清单中的每个独立设备/工序）
2. 估算覆盖所需的最少内容量：`最少字数 = 清单项数 × 150`（每项至少150字描述：安装位置、方法、要点、验收标准）
3. 与页数目标换算的字数对比：
   - 如果 `最少字数 ≤ 目标字数` → 正常生成，施工方案篇可充分展开
   - 如果 `最少字数 > 目标字数` → **必须告知用户**，给出两个选择：
     - 方案A：接受更多页数，确保施工方案全面覆盖所有清单项
     - 方案B：保持页数目标，但施工方案只能精简（明确告知哪些清单项会被合并描述）

**告知话术模板：**
```
⚠️ 内容覆盖分析：
   工程量清单包含 X 项施工内容，最少需要约 Y 字才能完整覆盖。
   您要求的 Z 页（约 W 字）不足以将所有清单项展开描述。
   - 方案A：放宽到约 N 页，确保每项施工内容都有详细方案
   - 方案B：保持 Z 页，施工方案按大类合并描述（如"道闸系统设备安装"统述，不逐项展开）
   建议选方案A。请问如何选择？
```

**注意：** 如果用户没有提供工程量清单，则跳过此分析，直接按页数目标规划。

### ⚠️ 严禁内容重复（最重要的规则）

生成内容时**绝对禁止出现重复段落**。同一句话、同一段话在文档中只能出现一次。

**常见的重复场景（必须避免）：**

| 场景 | 错误做法 | 正确做法 |
|------|---------|---------|
| 凑字数 | 把同一段话反复写多遍 | 每段内容必须唯一，宁可内容精炼 |
| 跨章节重复 | 各章节都写相同的"我公司将严格遵守..." | 每个章节写本节的特定内容，通用承诺只在总述写一次 |
| 首段模板化 | 每章开头都是"本项目是XXX，我公司将提供优质服务" | 每章直接切入主题，不重复项目介绍 |
| 末尾堆砌 | 文档末尾加几段"人员管理是...质量管理是...安全管理是..."的车轱辘话 | 末尾章节写完整后自然结束 |

**自检方法（生成完成后必须执行）：**

1. 逐段检查：任意两段文字含义或表述相似 → 删除重复，只保留一段
2. 检查文档末尾：如果末尾有大量看起来"似曾相识"的段落 → 全部删除
3. 如果感觉"好像在哪写过" → 这就是重复，删掉

**⚠️ 不视为重复的正常情况：**

以下情形即使在生成脚本的自动检测中被标记，也属于文档结构正常需要，不应删除：
- 文档开篇总述段落与第一章概述段落都提到项目名称——开篇是全文引言，章节开头是具体施工内容的引出，功能不同
- 不同章节出现相同的国家标准编号（如GB 50300-2013）——这是技术引用，不是内容重复
- 表格与正文中包含相同的数据指标——表格是汇总，正文是详细解释

**记住：用户一眼就能看出来重复内容，这是最严重的质量问题。宁可内容少而精，也绝不允许重复。**

### 步骤 2: 生成文档内容

根据用户提供的章节结构，逐章生成专业内容。

**内容质量标准（必须同时满足以下所有要求）：**

| 要求 | 说明 | 检查标准 |
|------|------|---------|
| **描述清晰** | 每段话逻辑清楚、条理分明，不使用模糊表述 | 读完后能明确知道"做什么、怎么做、标准是什么" |
| **内容具体** | 有具体数据、指标、方法名称、规范编号，不泛泛而谈 | 比如写质量标准要写"GB 50300-2013"而不是"国家规范" |
| **内容全面** | 从多角度、多层次展开，覆盖该主题的各个方面 | 管理措施应包含组织、制度、流程、检查、整改全链条 |
| **内容完整** | 每个章节有头有尾，有总述有分述有小结，不自相矛盾 | 表格完整、编号连续、引用有出处 |
| **内容翔实丰富** | 综合运用段落、表格、列表、流程图（用表格模拟）、图片占位符等多种表达方式 | 每个子章节至少含1个表格，重点子章节含2-3个表格；每章至少含1个列表 |
| **图片占位** | 施工平面图、流程图、架构图等无法用文字替代的内容，必须用图片占位符标注 | 每章至少1个图片占位符，见下方"图片占位符使用指引" |
| **图文结合** | 使用表格模拟流程图/组织架构图/流程图/饼图等视觉效果 | 见下方"图文结合示例" |

**图文结合示例（用表格模拟图形）：**

组织架构图用表格模拟：
```
┌─────────────────────────────────────────┐
│              项目经理                     │
├──────────┬──────────┬──────────────────┤
│ 技术负责人 │ 施工员    │ 安全员           │
├──────────┼──────────┼──────────────────┤
│ 电工班    │ 水暖班   │ 综合维修班        │
└──────────┴──────────┴──────────────────┘
```

→ 用表格实现：headers: ["技术负责人", "施工员", "安全员"], rows: [["电工班", "水暖班", "综合维修班"]]

流程图用表格模拟：
```
┌──────┐   →   ┌──────┐   →   ┌──────┐   →   ┌──────┐
│ 报修  │       │ 派单  │       │ 维修  │       │ 验收  │
└──────┘       └──────┘       └──────┘       └──────┘
```
→ 用表格实现：一行多列，每列一个步骤

**内容生成要点：**

- **严格遵循用户指定的章节标题和顺序**，一字不差
- **每章内容原创且不重复** — 每章的每段话必须唯一，禁止跨章节复制
- **内容充实**：每章至少 3-5 段实质性内容，不能只有标题
- **专业规范**：使用规范的行业术语
- **格式丰富**：适当使用表格、列表、分段来组织内容
- **紧扣项目**：内容要针对用户的项目特点，不能是通用模板（如用户项目是零星维修，就不能写成建筑施工）
- **工程量清单全覆盖**：施工方案章节中，工程量清单的每一项设备/工序都必须有对应的安装方法、技术参数或施工说明，不得遗漏。生成后用清单逐项核对

**不同类型项目的内容重点：**

| 项目类型 | 内容侧重点 |
|---------|-----------|
| 建筑施工 | 施工工艺、技术措施、质量安全、进度计划、资源配置 |
| 维修改造/零星维修 | 响应机制、人员配置、故障处理、快速反应、服务质量 |
| 服务采购项目 | 服务方案、管理团队、服务承诺、应急保障、售后服务 |
| 设备安装 | 安装工艺、调试方案、技术参数、验收标准 |

### 步骤 3: 按子章节字数目标生成内容（关键方法）

**核心思路：每个子章节（二级标题）写 3000-5000 字，用子章节数量来达到总字数目标。**

```
总字数目标 ÷ 4000 = 需要写的子章节数量（取中位值 4000）
```

> **实际经验值**：在实际生成中，每个子章节平均产出约 2400-3500 字。规划时应取保守值 3000-4000 字/子章节，而非 5000 字，这样规划的子章节数才足够。例：需要 55000 字 → 55000 ÷ 3500 ≈ 16 个子章节。

操作方法：

**1. 如果用户指定了总字数或总页数：**

先换算：`总字数 = 总页数 × 530`（每页约530汉字，保守估算）

然后按章节分配字数：
- 每个 **子章节**（二级标题）写 **3000-5000 字**
- 计算需要多少个子章节：`子章节数 = 总字数 ÷ 3500`（取保守值）
- 将子章节合理分配到各章（一级标题）中
- 每章至少1个子章节，重点章节多分配

**⚠️ 首轮字数自检（必须执行）**：

第一轮（或每轮）生成完毕、写入 temp.json 后，必须立即运行字数检查：

```bash
node .claude/skills/generate-construction-plan/scripts/generate-docx.js \
  --input temp.json --output _check.docx
```

检查输出的估算页数和完成度百分比：
- 完成度 ≥ 95% → 可以结束
- 完成度 80-95% → 该轮/下轮补充分配额外的子章节
- 完成度 < 80% → **必须在同一轮内追加内容**，不要等到全部章节写完再补救

**2. 如果用户没有指定字数：**

默认每个子章节写 3000-5000 字，根据章节复杂程度调整：
- 重点章节（施工技术措施、质量保证措施等）：3-4个子章节，共12000-20000字
- 一般章节：2个子章节，6000-10000字
- 简单章节：1个子章节，3000-5000字

**3. 每个子章节如何写足3000-5000字：**

```
┌─────────────────────────────────────────┐
│ 二级标题（如"一、质量管理体系"）        │  ~50字
│ 段落1：总体介绍该子章节内容              │  ~350字
│ 表格1：组织架构/职责分工/参数对照        │  ~200字
│ 段落2：具体措施/方法1（含数据指标）       │  ~400字
│ 段落3：具体措施/方法2（含规范编号）       │  ~400字
│ 表格2：流程步骤/质量标准/验收指标        │  ~200字
│ 段落4：实施要点/操作流程                 │  ~400字
│ 图片占位符：示意图/流程图/布置图          │  ~50字
│ 段落5：质量检查标准/验收依据             │  ~350字
│ 段落6：注意事项/常见问题防范             │  ~350字
│ 表格3：检查清单/对比表（可选，重点子章节）│  ~200字
│ ...                                      │
│ 合计                                     │  ~3500字
└─────────────────────────────────────────┘
```

> **充实技巧**：每个段落不只是写"是什么"，更要写"怎么做、标准是什么、谁来检查、不达标怎么办"——把管理闭环写完整，自然就有字数了。

**4. 关键规则：**

- **每个子章节的内容必须完全独立、绝不重复** — 从不同角度/层面展开
- 如果发现某个子章节写不够字数 → 增加更深层次的细节内容（扩大具体措施、增加数据表格、补充案例场景），不要往回找补
- 如果用户给的章节结构里没有二级标题 → 在content中使用heading level 2来划分子章节
- **字数只是参考，质量优先** — 写不够就如实说，**绝不通过重复来凑**

### 步骤 3b: 处理超大型文档（100页以上 / 50000字以上）

对于超大型文档，Claude 无法一次生成全部内容。必须采用 **分批增量写入** 的方式。

**Token 成本参考（提前告知用户）：**

| 规模 | 页数 | 约需轮数 | 总Output | 总Input | 合计 |
|------|------|---------|---------|---------|------|
| 小型 | ~10页 | 1轮 | ~5.5K | ~3K | ~8.5K |
| 中型 | ~50页 | 2轮 | ~27K | ~6K | ~33K |
| 大型 | ~100页 | 4轮 | ~55K | ~12K | ~67K |
| 超大型 | ~500页 | 19轮 | ~275K | ~57K | ~332K |
| 巨型 | ~1000页 | 37轮 | ~550K | ~111K | ~661K |

> 注：1页 ≈ 530汉字 ≈ 530 tokens。按每轮生成3-5章（~15-20K tokens）估算。

**优化策略（减少Token消耗）：**

| 策略 | 做法 | 效果 |
|------|------|------|
| 增大每轮批次 | 每轮生成3-5章而非2-3章 | 轮数减半，总token省30-40% |
| 精简已有内容回读 | 回读时只读章节标题和字数，不读全文 | 每轮input减少50-70% |
| 一次成型 | 如果100页内，尽量一轮完成全部章节 | 避免回读开销 |

#### 标准操作流程（关键：必须用 Write→Node 方式，禁止用超长 bash -e）

**第1轮：创建初始 JSON 文件**

1. 用 `Write` 工具写一个 `create-initial.js` 脚本，内含完整的 chapters 构建逻辑
2. 运行 `node create-initial.js` 创建 temp.json
3. **立即验证 JSON 合法性**（见步骤0的验证命令）
4. **立即运行** `node generate-docx.js --input temp.json --output _check.docx` 检查字数，如不足目标80%则在同一轮扩充

**第 N 轮（N≥2）：追加新章节**

1. 用 `Write` 工具写一个 `append-round-N.js` 脚本
2. 脚本内容使用以下标准化模板：

```javascript
// append-round-N.js — 追加第N批章节到 temp.json
const fs = require('fs');
const data = JSON.parse(fs.readFileSync('temp.json', 'utf8'));
console.log('现有章节:', data.chapters.length, '个');

// === 在此定义新章节 ===
const newChapter = {
  title: '第X章、XXX',
  level: 1,
  content: [
    // ... 在此填写内容，注意使用 “” 而非 ""
  ]
};
// ======================

data.chapters.push(newChapter);  // 或 push(...newChapters)
fs.writeFileSync('temp.json', JSON.stringify(data, null, 2), 'utf8');

// 验证
try {
  JSON.parse(fs.readFileSync('temp.json', 'utf8'));
  console.log('追加成功！总章节:', data.chapters.length);
} catch(e) {
  console.log('JSON错误:', e.message);
}
```

3. 运行 `node append-round-N.js`
4. **立即验证 JSON 合法性**

**最后一轮：生成最终文档**

```bash
node .claude/skills/generate-construction-plan/scripts/generate-docx.js \
  --input temp.json \

```

### 步骤 4: 构建 JSON 数据

组装为以下 JSON 格式：

```json
{
  "projectName": "XX项目",
  "subtitle": "施工组织服务方案",
  "companyName": "XXXX（如有）",
  "date": "2026年6月",
  "targetWordCount": 8000,
  "targetWordsPerSection": 3500,
  "outputName": "输出文件名",
  "chapters": [
    {
      "title": "第一章、施工管理目标",
      "level": 1,
      "content": [
        { "type": "paragraph", "text": "正文内容...", "format": { "firstLineIndent": 2 } },
        { "type": "heading", "text": "子标题", "level": 2 },
        { "type": "list", "items": ["项1", "项2"], "listType": "bullet" },
        { "type": "table", "headers": ["列1", "列2"], "rows": [["A", "B"]] },
        { "type": "pageBreak" }
      ]
    }
  ]
}
```

**content 单元类型：**

| type | 必填 | 说明 |
|------|------|------|
| `paragraph` | `text` | 段落，支持 `format.firstLineIndent`(缩进字符)、`format.bold`、`format.alignment`(left/center/both) |
| `heading` | `text`, `level` | 子标题，level 2-3 |
| `list` | `items`(string[]) | 列表，`listType`: "bullet"序号 / "number"数字 |
| `table` | `headers`, `rows` | 表格，表头自动加蓝底；可选 `caption` 生成"表X-Y 标题" |
| `image` | `base64`, `width`, `height` | 图片，base64编码（不含data:前缀），width/height为像素 |
| `placeholder` | `title`, `width`, `height`, `caption` | 图片占位符，生成带虚线边框的占位框+图名（无需base64） |
| `pageBreak` | — | 分页符 |

**image 类型说明：**
```json
{ "type": "image", "base64": "iVBORw0KGgo...", "width": 500, "height": 300, "caption": "图1-1 管理体系流程图" }
```
- `base64`：图片的 base64 编码数据（不含 data:image/xxx;base64, 前缀）
- `width` / `height`：图片在文档中的显示尺寸（像素）
- `caption`（可选）：图片标题，显示在图片下方

#### ⚠️ 图片占位符使用指引（必须遵守）

施工组织设计需要大量图形：施工平面布置图、组织架构图、进度横道图、系统拓扑图、流程图等。由于无法提供真实的 base64 图片数据，使用 `placeholder` 类型来生成带边框的图片占位框。

**每个章节至少插入1-2个图片占位符**，以下位置为必插位置：

| 位置 | 图片类型 | 示例 caption |
|------|---------|-------------|
| 第一章施工方案 | 施工平面布置图 | "图1-1 施工现场平面布置图" |
| 第一章施工方案 | 施工流程图（用表格模拟） | "图1-2 出入口道闸系统施工流程图" |
| 第一章施工方案 | 系统拓扑图 | "图1-3 系统网络拓扑图" |
| 第二章技术组织措施 | 质量管理体系组织架构图 | "图2-1 质量管理体系组织架构图" |
| 第二章技术组织措施 | 工序质量控制流程图 | "图2-2 工序质量控制流程图" |
| 第三章劳动力安排 | 劳动力投入柱状图/曲线图 | "图3-1 各阶段劳动力投入计划图" |
| 第三章劳动力安排 | 施工进度横道图（甘特图） | "图3-2 施工进度计划横道图" |
| 第四章应急预案 | 应急组织体系图 | "图4-1 应急指挥组织体系图" |
| 第四章应急预案 | 应急响应流程图 | "图4-2 应急响应流程图" |

**placeholder JSON 格式：**

```json
{ "type": "placeholder", "title": "此处插入施工平面布置图", "width": 500, "height": 350, "caption": "图1-1 施工现场平面布置图" }
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `title` | ✅ | 占位框中显示的提示文字，描述此处应插入什么图 |
| `width` | 否 | 占位框宽度（像素），默认 500 |
| `height` | 否 | 占位框高度（像素），默认 350 |
| `caption` | ✅ | 图片标题，显示在占位框下方，格式为"图X-Y 图片名称" |

**效果：** 在 Word 文档中生成一个带虚线边框的矩形占位框，框内居中显示提示文字，框下方显示图片标题。用户打开文档后可在占位框位置插入实际图片。

> 注意：组织架构图、流程图等能用表格模拟的图形，优先用 `table` 模拟（跨行跨列），不要用 placeholder。placeholder 只用于无法用表格表达的图（如平面布置图、系统拓扑图、实景照片等）。
